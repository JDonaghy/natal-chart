1. Project Overview

A browser-first, serverless birth chart application that computes and renders traditional Western natal charts entirely on the client side. The app will be hosted on GitHub Pages, protected against abuse with Cloudflare, and built with a React codebase that shares logic with a future React Native mobile app.

2. Guiding Principles

  • Serverless by default. No persistent server. All chart math runs in the browser via WASM. Only a Cloudflare Worker is used as a thin proxy for geocoding (to hide the API key and apply rate limits).

  • One codebase, two platforms. React (web) + React Native (mobile) share all business logic, calculation hooks, and data models. Only rendering differs.

  • Privacy-respecting. Birth data never leaves the device except for geocoding (city → lat/lng). No accounts, no data storage by default in v1.

  • Bot protection without a backend. Cloudflare Turnstile handles human verification at the edge for free, with zero server infrastructure.

  • Open source, responsibly. API keys live only in Cloudflare Worker secrets — never in the repo.

5. Calculation Engine (packages/core)

5.1 Swiss Ephemeris WASM

The Swiss Ephemeris is the gold standard for astrological calculation, used by professional software worldwide. The npm package swisseph-wasm provides a pre-compiled WebAssembly build that runs in any modern browser.



pnpm add swisseph-wasm



Critical: swisseph-wasm requires ephemeris data files (*.se1) to be co-located with the WASM binary and served with the correct MIME type. Vite's public/ folder is the right place. Validate this early — it is the most common failure mode.



  • Planets computed: Sun, Moon, Mercury, Venus, Mars, Jupiter, Saturn, Uranus, Neptune, Pluto, North Node, Chiron

  • Ascendant and Midheaven computed via swe_houses()

  • Default house system: Placidus (most common in Western astrology). Expose as a user setting later.

  • All times must be converted to Julian Day Number (JDN) in UTC before passing to the library.

5.2 Core Module API

Expose a clean TypeScript API from packages/core so neither web nor mobile touches swisseph-wasm directly:



interface BirthData {

dateTimeUtc: Date; // already converted from local time

latitude: number; // decimal degrees

longitude: number; // decimal degrees

houseSystem: 'P' | 'W' | 'K'; // Placidus | Whole Sign | Koch

}



interface ChartResult {

planets: PlanetPosition[]; // sign, degree, house, retrograde

houses: HouseCusp[]; // 12 cusps in ecliptic longitude

angles: Angles; // ASC, MC, DSC, IC

aspects: Aspect[]; // pairs + orb + applying/separating

}



export async function calculateChart(data: BirthData): Promise<ChartResult>

6. Geocoding Proxy (packages/worker)

6.1 Why a Proxy

Geocoding APIs require a key. Exposing that key in browser JavaScript would allow anyone to exhaust your quota. A Cloudflare Worker solves this: the key lives in Worker Secrets (encrypted, not in git), and the Worker enforces rate limits before forwarding requests to OpenCage.

6.2 Recommended Service: OpenCage

OpenCage Geocoding API is well-suited for this use case: 2,500 free requests/day, no credit card required for the free tier, and good coverage for historical birth locations.



GET https://api.opencagedata.com/geocode/v1/json

?q={city+country}&key={SECRET}&limit=5&no_annotations=1



6.3 Worker Logic

  • Accept POST /geocode from the frontend with { query: string }

  • Validate Cloudflare Turnstile token (see Section 8)

  • Apply rate limit: max 10 requests per IP per minute using Cloudflare's built-in rate limiting

  • Forward to OpenCage, return top 5 results (name, lat, lng, country)

  • Cache responses in KV for 30 days — birth cities don't move



Set CORS headers to allow only your GitHub Pages domain. This prevents other sites from using your worker as a free geocoding relay.



7. GitHub Pages Deployment

7.1 Vite Configuration

GitHub Pages serves sites at https://{username}.github.io/{repo-name}/. Vite must be told the base path:



// vite.config.ts

export default defineConfig({

base: '/natal-chart/', // matches your repo name

build: { outDir: 'dist' }

})



7.2 GitHub Actions CI/CD

Create .github/workflows/deploy.yml. On every push to main: install dependencies, run tests, build, deploy to the gh-pages branch.



on: push: branches: [main]

jobs:

deploy:

runs-on: ubuntu-latest

steps:

- uses: actions/checkout@v4

- uses: pnpm/action-setup@v3

- run: pnpm install

- run: pnpm --filter web build

- uses: peaceiris/actions-gh-pages@v4

with: { github_token: ${{ secrets.GITHUB_TOKEN }}, publish_dir: packages/web/dist }



7.3 SPA Routing

GitHub Pages does not support client-side routing out of the box. Use the 404.html redirect trick: copy index.html to 404.html in the build output so that deep links reload the app rather than returning a 404. Alternatively, use hash-based routing (React Router with createHashRouter) to avoid the issue entirely — recommended for v1.



8. Bot Protection

8.1 Cloudflare Turnstile

Turnstile is Cloudflare's privacy-preserving alternative to reCAPTCHA. It is free with no usage limits and requires no Cloudflare-hosted domain — it works with GitHub Pages.



  • Frontend: embed the Turnstile widget using @marsidev/react-turnstile

  • On solve, the widget returns a short-lived token

  • Send the token with every geocoding request to the Worker

  • Worker validates the token server-side with Cloudflare's siteverify API before forwarding to OpenCage



// Worker validation

const res = await fetch('https://challenges.cloudflare.com/turnstile/v0/siteverify', {

method: 'POST',

body: JSON.stringify({ secret: env.TURNSTILE_SECRET, response: token })

})



Turnstile tokens expire after 5 minutes and can only be validated once. For chart calculation (which is 100% client-side WASM), no bot protection is needed — only the geocoding endpoint needs protection.

8.2 Rate Limiting

Configure Cloudflare Rate Limiting rules on the Worker route: 10 requests/minute per IP for geocoding, 60/minute per IP overall. This is configured in the Cloudflare dashboard, not in code.



9. Chart Wheel Rendering

9.1 Approach: SVG in React

Render the natal chart wheel as an SVG element directly in React. SVG gives precise control over every line, glyph, and arc. Avoid canvas for this — SVG is easier to theme, scale, and later adapt to React Native SVG.

9.2 Visual Style: Classic / Traditional

Execute a parchment-and-gold aesthetic consistent with vintage printed ephemerides:



  • Background: warm parchment (#F5F0E8) or near-white cream

  • Wheel rings: multiple concentric circles with fine gold stroke (#B8860B or #D4AF37)

  • Zodiac band (outermost ring): alternating pale fills per sign, classic glyph symbols

  • House lines: thin gold radial spokes from center to zodiac ring

  • Planet glyphs: Unicode astrological symbols (☉ ☽ ♂ ♃ etc.) in a serif font

  • Aspect lines (innermost circle): color-coded thin lines — blue trines, red squares, green sextiles

  • Typography: a serif or old-style font (e.g. IM Fell English, Crimson Text, or Cormorant) for degree labels

9.3 Rendering Layers (SVG groups)

Structure the SVG with explicit <g> groups in z-order:



1. Background circle (parchment fill)

2. Aspect lines group (innermost, below everything)

3. House division lines

4. Zodiac sign band

5. Degree tick marks

6. Sign glyphs + labels

7. Planet glyph group (topmost, must not overlap)

8. ASC / MC angle markers



Planet glyph collision detection is the hardest rendering problem. When two planets are within ~5° of each other, their glyphs overlap. Implement a simple angular offset algorithm: if two planets are within a threshold, nudge one outward along the radius.

9.4 Coordinate System

Astrology wheels conventionally place 0° Aries at the 9 o'clock position (left), with degrees increasing counter-clockwise. Convert ecliptic longitude to SVG screen angle as:



// svgAngle: 0 = right (3 o'clock), increases clockwise

// eclLon: 0 = Aries, increases counter-clockwise from 9 o'clock

svgAngle = (180 - eclLon + ascendantDegree) % 360



The Ascendant always sits on the left (9 o'clock). Rotate the entire wheel so ASC aligns there.



10. State Management

Keep it simple for v1. No Redux or Zustand needed.



  • Use React Context for the current ChartResult (available app-wide)

  • Use useState / useReducer within the birth data form

  • Persist the last chart to localStorage so the user's chart survives a page refresh

  • URL hash encoding of birth data (base64 JSON) enables shareable chart links without a database

12. Mobile Path (React Native)

The mobile app shares packages/core entirely unchanged. The only differences are rendering and navigation:



  • Use react-native-svg in place of browser SVG — the chart component will need a thin adapter

  • Use Expo for the easiest iOS/Android build pipeline

  • Geocoding calls the same Cloudflare Worker endpoint

  • Consider Expo EAS Build for CI/CD to App Store / Google Play



Do not start React Native until the web app is stable. The shared core package is the bridge — keep it clean and dependency-free (no browser APIs, no React).

14. Key Risks & Mitigations



Risk

Likelihood

Mitigation

swisseph-wasm fails to load ephemeris data files

Medium

Test this first, before any UI work. File paths and MIME types are the usual culprits.

Planet glyph overlap on crowded charts

High

Build collision offset algorithm early; it affects every chart.

OpenCage quota exhaustion

Low

Cloudflare KV caching eliminates repeat queries for the same city.

GitHub Pages base-path routing breaks

Medium

Use hash router (#/) in v1 to sidestep entirely.

Turnstile blocks legitimate users

Low

Turnstile's challenge-free mode passes most real users silently.



15. Getting Started — Recommended Order



  1. Create GitHub repo, initialise pnpm workspace, add packages/core, packages/web, packages/worker

  2. Spike: load swisseph-wasm in a bare Vite app and compute the Sun position for a known date. Validate against an online chart service.

  3. Implement packages/core: BirthData → ChartResult, with full planet, house, and aspect output

  4. Build the birth data form (date, time, city search) with the geocoding worker behind Turnstile

  5. Build the SVG chart wheel — start with circles, houses, and sign glyphs before adding planets

  6. Add planet glyphs with collision detection, then aspect lines

  7. Set up GitHub Actions deploy workflow and verify live on GitHub Pages

  8. Add localStorage persistence and shareable URL encoding

  9. Polish UI to the classic/traditional aesthetic, responsive layout, accessibility



Steps 1–3 are best done in a single Claude Code session. Step 2 (the swisseph-wasm spike) is the highest-risk item and should be green before any other work proceeds.
